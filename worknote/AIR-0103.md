## Goal
- Keep longform recommendation claims on the longform worker path so claim success cannot send a normal longform worker into `/music-plan`.

## Context
- `AIR-0102` documented the current longform flow and called out mode-routing ambiguity around `/script-plan` and `/music-plan`.
- The user clarified the required rule:
  - longform topic claim -> project creation -> `/script-plan`
  - `/music-plan` only for real `longform_music` internal mode
  - standard online longform workers must not enter the music workflow

- Code review:
  - `app/routers/pages.py`
  - `app/routers/user_topics.py`
  - `templates/pages/projects.html`
  - `templates/base.html`
- Runtime validation with `TestClient`:
  - longform project `GET /script-plan?project_id={id}` returns `200`
  - music project `GET /script-plan?project_id={id}` redirects to `/music-plan?project_id={id}`
  - longform project `GET /music-plan?project_id={id}` redirects to `/script-plan?project_id={id}`
  - standard membership `GET /music-plan?project_id={music_id}` redirects to `/projects`

## Result
- `/script-plan` now resolves mode from the selected project first, not only the global app mode.
- `/music-plan` now rejects non-music projects and routes them back to `/script-plan`.
- Standard memberships are blocked from entering `/music-plan`, which keeps normal online longform workers out of the music workflow.
- `LONGFORM_USER_FLOW.md` now records this rule explicitly.

## Known Issues
- Other longform/music routes still have some global-mode branching and should be audited separately.
- Live `longform_music` recommendation verification remains blocked by the deployed Supabase category schema.
